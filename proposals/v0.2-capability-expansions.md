# OpenLang v0.2 Capability Expansions — Proposal

Three additions to the v0.1 spec. Each is backward-compatible: a v0.1 speaker that does not understand these features will hit `~unk` and can request `~def` or fall back to `~L1` as normal.

---

## 1. Concurrency Primitives

### Problem

`^par` and `^seq` cover basic parallel/sequential dispatch but cannot express real orchestration patterns: exclusive resource access, inter-task communication, bounded waits, or structured fork/join with named handles.

### New Control Flow Tokens

| Token | Meaning | Category |
|-------|---------|----------|
| `^frk` | Fork — spawn named task | Control Flow |
| `^jn` | Join — await named task(s) | Control Flow |
| `^lk` | Lock — acquire named mutex | Control Flow |
| `^ulk` | Unlock — release named mutex | Control Flow |
| `^ch` | Channel — declare named channel | Control Flow |
| `^tx` | Transmit — send to channel | Control Flow |
| `^rx` | Receive — read from channel | Control Flow |
| `^tmo` | Timeout — wrap operation with deadline | Modifier |

### Syntax

#### Fork / Join

```
-- Fork two named tasks, join both, collect results
^frk:t1 !fnd @fs {p:"src/**/*.ts" rgx:"TODO"}
^frk:t2 !fnd @fs {p:"docs/**/*.md" rgx:"FIXME"}
^jn [t1,t2] ->results

-- Join with timeout (seconds)
^jn [t1,t2] ^tmo:30 ->results | ^el >fl {err:timeout}
```

`^frk:name` spawns a named concurrent task. `^jn [names]` blocks until all complete. Results are collected as a list in fork order.

#### Lock / Unlock

```
-- Exclusive access to shared resource
^lk:db_write
!wr @db {tbl:users row:{id:1 name:"x"}}
!wr @db {tbl:logs row:{action:update}}
^ulk:db_write
```

Lock names are arbitrary strings. A lock is scoped to the agent group in the current session. Attempting to acquire a held lock blocks until released.

#### Channels

```
-- Declare a channel
^ch:results ::int

-- Producer (in a forked task)
^frk:scanner ^ea {src:files} ->f {
  ?fnd @fs {p:f rgx:"ERR"} ->n
  ^tx:results {v:n}
}

-- Consumer
^rx:results ->count lmt:10
```

Channels are typed optionally with `::`. `^tx` sends a value, `^rx` reads. Channels are unbuffered by default. Add `buf:N` to `^ch` for buffered:

```
^ch:results ::int buf:50
```

#### Timeout

`^tmo:N` (seconds) can wrap any blocking operation:

```
^wt {cond:ready} ^tmo:60 | ^el >fl {err:timeout}
^rx:data ^tmo:5 ->val | ^el >fl {err:ch_timeout}
```

### Full Example — Fan-Out/Fan-In

```
{id:m1 from:orchestrator to:worker_pool
 ~L2
 ^ch:findings ::map buf:100
 ^frk:w1 !fnd @fs {p:"svc-a/src/**" rgx:"vulnerability"} ->r; ^tx:findings {v:r}
 ^frk:w2 !fnd @fs {p:"svc-b/src/**" rgx:"vulnerability"} ->r; ^tx:findings {v:r}
 ^frk:w3 !fnd @fs {p:"svc-c/src/**" rgx:"vulnerability"} ->r; ^tx:findings {v:r}
 ^jn [w1,w2,w3] ^tmo:120
 ^rx:findings ->all_results
 !snd @net {to:dashboard body:all_results}
}
```

### Backward Compatibility

All new tokens are under the `^` sigil. A v0.1 agent will `~unk` on `^frk`, `^jn`, etc. The sender can then either:
- `~def` the tokens inline
- Fall back to `^par` / `^seq` which v0.1 understands (losing named-task semantics)

## 2. Qualifier / Adjective Tokens

### Problem

The vocabulary is action-heavy. Filtering, sorting, and state description require dropping to L1 or abusing parameter blocks. Short qualifier tokens fix this.

### New Tokens — Qualifiers Category

| Token | Meaning | Mnemonic |
|-------|---------|----------|
| `rcn` | recent | recent |
| `lrg` | large | large |
| `sml` | small | small |
| `chg` | changed | changed |
| `stl` | stale | stale |
| `nw` | new | new |
| `old` | old | old |
| `act` | active | active |
| `idl` | idle | idle |
| `fld` | failed | failed |
| `hlt` | healthy | healthy |
| `hot` | hot/frequently accessed | hot |
| `cld` | cold/rarely accessed | cold |

### Syntax

Qualifiers appear as modifiers on scopes, targets, or inside parameter blocks:

```
-- Find recently changed TypeScript files
?fnd @fs chg rcn {p:"src/**/*.ts"}

-- List large files
?fnd @fs lrg {p:"**/*" min:10mb}

-- Show only active processes
?fnd @proc act

-- Delete stale cache entries
!del @mem stl {k:cache}

-- Query failed tasks from the last run
?fnd @proc fld rcn

-- Health check — find unhealthy services
?fnd @net {svc:"*"} ->lst | ^ea ->s ?vfy @net {svc:s} ->st | ^if {st:fld} #rpt {s:fld} ^el #rpt {s:hlt}
```

### Qualifier Stacking

Qualifiers compose left-to-right as successive filters:

```
-- New and large files
?fnd @fs nw lrg {p:"uploads/**"}

-- Old idle processes
?fnd @proc old idl ->lst
```

### In Status Reporting

Qualifiers describe state in `#` data blocks:

```
#sys {cpu:act mem:72% disk:hlt net:idl}
#cache {entries:450 stl:38 hot:12 cld:400}
```

### Backward Compatibility

Qualifiers are new vocabulary, not new grammar. A v0.1 agent that doesn't recognize `rcn` will `~unk {tok:"rcn" req:def}`, and the sender responds with `~def {tok:"rcn" means:"recent/recently modified"}`. No structural changes required.

## 3. Versioned Grammar Negotiation

### Problem

The current `~hello` handshake declares `ver:0.1` and `~cap` with supported levels and extensions, but provides no mechanism to:
1. Enumerate what grammar features a version adds
2. Negotiate a common subset when agents differ in version
3. Gracefully degrade specific features

### Design

#### Version Feature Sets

Each version declares a feature set identifier. Features are cumulative — a higher version includes all lower features.

```
-- v0.1 features (implicit baseline)
v0.1 := {flow:[if,lp,ea,par,seq,wt,rt,br,ct] vocab:base}

-- v0.2 features (this proposal)
v0.2 := v0.1 + {flow:[frk,jn,lk,ulk,ch,tx,rx,tmo] qual:[rcn,lrg,sml,chg,stl,nw,old,act,idl,fld,hlt,hot,cld]}
```

#### Extended Handshake

The `~hello` message gains a `feat` field listing supported feature groups:

```
-- v0.2 agent
~hello {id:bot1 ver:0.2 ~cap {L:[1,2,3] ext:[fs,git,sh] feat:[conc,qual]}}

-- v0.1 agent (no feat field = baseline only)
~hello {id:bot2 ver:0.1 ~cap {L:[1,2] ext:[fs,db]}}
```

Feature group identifiers:

| Feature ID | Covers | Since |
|------------|--------|-------|
| `base` | v0.1 full spec (implicit, always present) | v0.1 |
| `conc` | Concurrency primitives (frk/jn/lk/ulk/ch/tx/rx/tmo) | v0.2 |
| `qual` | Qualifier tokens (rcn/lrg/sml/chg/stl/nw/old/act/idl/fld/hlt/hot/cld) | v0.2 |

#### Negotiation Protocol

After handshake, the higher-version agent computes the intersection and declares the session grammar:

```
-- Bot1 (v0.2) sees bot2 is v0.1, proposes session terms
~sess {ver:0.1 feat:[base] degrade:[conc,qual]}

-- Bot2 acknowledges
~ack {id:sess}
```

The `degrade` field lists features the higher-version agent will avoid. This is informational — the sender commits to not using those tokens without inline `~def`.

#### Selective Feature Opt-In

A v0.1 agent that has been taught features via `~def` can declare them:

```
-- v0.1 agent that learned concurrency tokens
~hello {id:bot2 ver:0.1 ~cap {L:[1,2] ext:[fs,db] feat:[conc]}}
```

This signals: "I'm on v0.1 grammar but I understand concurrency primitives." The session negotiation then includes `conc`:

```
~sess {ver:0.1+ feat:[base,conc] degrade:[qual]}
```

The `0.1+` notation means: base grammar is v0.1, with listed extensions active.

#### Inline Feature Introduction

If mid-conversation a feature becomes needed that was degraded, the sender can introduce it:

```
~feat:conc {
  ~def {tok:"^frk" means:"fork named task: ^frk:name <body>"}
  ~def {tok:"^jn" means:"join tasks: ^jn [names] ->result"}
  ~def {tok:"^tmo" means:"timeout in seconds: ^tmo:N"}
}
~sess:upd {feat+:[conc]}
```

After `~sess:upd`, the token is available for the rest of the session. The receiver must `~ack` or `~rej`:

```
~ack {id:sess:upd}          -- accepted, conc now active
~rej {id:sess:upd why:unsupported}  -- declined
```

### Full Handshake Example

```
-- Agent A (v0.2) initiates
{id:h1 from:agentA to:agentB
 ~hello {id:agentA ver:0.2 ~cap {L:[1,2,3] ext:[fs,git,sh,net] feat:[conc,qual]}}
}

-- Agent B (v0.1) responds
{id:h2 from:agentB to:agentA
 ~hello {id:agentB ver:0.1 ~cap {L:[1,2] ext:[fs,git]}}
}

-- Agent A proposes session (auto-negotiated)
{id:h3 from:agentA to:agentB
 ~sess {ver:0.1 feat:[base] degrade:[conc,qual] L:2 ext:[fs,git]}
}

-- Agent B accepts
{id:h4 from:agentB to:agentA
 ~ack &h3
}

-- Later, Agent A needs fork/join and introduces it
{id:m5 from:agentA to:agentB
 ~feat:conc {
   ~def {tok:"^frk" means:"fork named concurrent task"}
   ~def {tok:"^jn" means:"join/await named tasks"}
   ~def {tok:"^tmo" means:"timeout wrapper (seconds)"}
 }
 ~sess:upd {feat+:[conc]}
}

{id:m6 from:agentB to:agentA
 ~ack &m5
}

-- Now Agent A can use concurrency tokens freely
{id:m7 from:agentA to:agentB
 ^frk:scan !fnd @fs {p:"src/**" rgx:"TODO"} ->lst
 ^frk:test !tst @sh {cmd:"vitest"} ->rpt
 ^jn [scan,test] ^tmo:60 ->results
}
```

### Backward Compatibility

- The `feat` field in `~hello` is optional. Absence means `feat:[base]`.
- The `~sess` and `~sess:upd` messages are new but use existing `~` meta sigil. A v0.1 agent that doesn't understand `~sess` will `~unk` it, at which point the v0.2 agent knows to stay on pure v0.1.
- The `~feat` block is syntactic sugar over multiple `~def` calls, which v0.1 already supports.
- No existing tokens change meaning. All additions are purely additive.

## Summary of All New Tokens

### Control Flow (`^` sigil)

| Token | Args | Meaning |
|-------|------|---------|
| `^frk:name` | body | Fork named task |
| `^jn` | `[names]` | Join/await tasks |
| `^lk:name` | — | Acquire mutex |
| `^ulk:name` | — | Release mutex |
| `^ch:name` | `::type buf:N` | Declare channel |
| `^tx:name` | `{v:val}` | Send to channel |
| `^rx:name` | `->var` | Receive from channel |
| `^tmo:N` | — | Timeout (seconds) |

### Qualifiers (vocabulary)

| Token | Meaning | Token | Meaning |
|-------|---------|-------|---------|
| `rcn` | recent | `lrg` | large |
| `sml` | small | `chg` | changed |
| `stl` | stale | `nw` | new |
| `old` | old | `act` | active |
| `idl` | idle | `fld` | failed |
| `hlt` | healthy | `hot` | hot |
| `cld` | cold | | |

### Meta (`~` sigil)

| Token | Args | Meaning |
|-------|------|---------|
| `~sess` | `{ver feat degrade L ext}` | Declare session grammar |
| `~sess:upd` | `{feat+:[...]}` | Add features mid-session |
| `~feat:id` | `{~def ...}` | Introduce feature group with definitions |
| `~rej` | `{id why}` | Reject proposal |
| `feat` | (in `~cap`) | List of supported feature group IDs |

### Version Feature Groups

| ID | Contents | Since |
|----|----------|-------|
| `base` | All v0.1 tokens and grammar | v0.1 |
| `conc` | ^frk ^jn ^lk ^ulk ^ch ^tx ^rx ^tmo | v0.2 |
| `qual` | rcn lrg sml chg stl nw old act idl fld hlt hot cld | v0.2 |
