# OpenLang v0.2 Proposals — Syntax/Grammar Fixes

Three targeted fixes for issues identified during v0.1 testing. Each is backward-compatible: v0.1 messages remain valid.

---

## 1. Block Scoping for Deep Nesting

### Problem

L2 uses `{}` for both parameter blocks and implicit nesting (control flow bodies, compositions). At depth 3+, brace matching becomes ambiguous:

```
^if {x>0} {^if {y>0} {^lp {n:3} {!mod @fs {p:"a.ts" v:x}}}}
```

Seven closing braces. No visual or structural cue about which `}` closes what.

### Proposal: `<<` / `>>` Block Delimiters

Introduce `<<` `>>` as explicit block-scope delimiters for compound statements. `{}` remains for parameter blocks only.

| Token | Role |
|-------|------|
| `{}` | Parameter block (unchanged) |
| `<< >>` | Block scope — wraps multi-statement or nested control flow bodies |

Rules:
- `<< >>` is **optional**. Single-expression bodies still work with `{}`.
- Use `<< >>` when a body contains nested control flow or multiple statements.
- `{}` inside `<< >>` is always a parameter block, never a body. Removes ambiguity.

### Before (v0.1 — ambiguous)

```
^if {x>0} {^if {y>0} {^lp {n:3} {!mod @fs {p:"a.ts" v:x}; !run @sh {cmd:"lint"}}}}
```

### After (v0.2 — scoped)

```
^if {x>0} <<
  ^if {y>0} <<
    ^lp {n:3} <<
      !mod @fs {p:"a.ts" v:x}
      !run @sh {cmd:"lint"}
    >>
  >>
>>
```

### Compact form (shallow nesting, no change needed)

```
^if {x>0} {!run @sh {cmd:"build"}} ^el {!run @sh {cmd:"clean"}}
```

This still works. `<< >>` is additive.

### Mixed example

```
^ea {src:files} ->f <<
  ?rd @fs {p:f} ->content
  ^if {content.sz>1000} {!mod @fs {p:f trunc:true}} ^el <<
    !fmt @fs {p:f}
    !vfy @fs {p:f}
  >>
>>
```

### Grammar addition

```
block := '{' expr '}' | '<<' stmt* '>>'
```

## 2. L3 Period Escaping for Literal Periods

### Problem

L3 bytecode is period-delimited: `sigil.scope.action.params...`

Parameters containing literal periods break parsing:

```
Q.fs.fnd.app.config.ts.rec
```

Is that `app` / `config` / `ts` / `rec` (four fields)? Or `app.config.ts` / `rec` (filename + modifier)?

### Proposal: Backtick Quoting

Wrap any L3 field containing a literal period in backticks:

```
`value.with.periods`
```

Rules:
- Backtick-quoted fields are treated as a single opaque token.
- Only required when a field contains `.` — all other fields are unchanged.
- Backticks do not appear in the resolved value (they are delimiters, not content).
- To include a literal backtick in a value, double it: ``` `` ```.

### Before (v0.1 — broken)

```
Q.fs.fnd.app.config.ts.rec
C.fs.del.dist/server.bundle.js
R.ok.1.[app.config.ts:12]
```

### After (v0.2 — unambiguous)

```
Q.fs.fnd.`app.config.ts`.rec
C.fs.del.`dist/server.bundle.js`
R.ok.1.[`app.config.ts`:12]
```

### Unquoted fields (no change)

```
Q.fs.fnd.*.ts.rec.3
R.ok.3.[a:5,b:12,c:3]
```

No backticks needed when fields contain no periods. Full backward compatibility.

### Grammar addition

```
l3_field := backtick_field | raw_field
backtick_field := '`' [^`]+ '`'
raw_field := [^.]+
l3_stmt := l3_field ('.' l3_field)*
```

## 3. Negation / Exclusion Modifier

### Problem

No way to express "NOT X" without dropping to L1 or writing awkward filter chains:

```
-- v0.1 workaround: find all .ts, then manually filter
?fnd @fs {p:"src/**" rgx:"TODO"} ->lst | ^ea ->f ^if {f.ext != "test.ts"} {!rd @fs {p:f}}
```

### Proposal: `!~` Negation Prefix (within params) and `neg` Modifier

Two complementary mechanisms:

#### a) `!~` prefix — negates a single parameter value

Place `!~` directly before a parameter value to negate/exclude it.

| Expression | Meaning |
|------------|---------|
| `{p:"*.ts"}` | match `*.ts` |
| `{p:!~"*.test.ts"}` | exclude `*.test.ts` |
| `{rgx:!~"TODO"}` | lines NOT matching `TODO` |
| `{ext:!~[js,jsx]}` | extensions not in `[js, jsx]` |

#### b) `neg` modifier — negates an entire condition in control flow

| Expression | Meaning |
|------------|---------|
| `^if {cond}` | if cond is true |
| `^if neg {cond}` | if cond is NOT true |

### Before (v0.1 — workaround)

```
-- find .ts files excluding tests, excluding node_modules
?fnd @fs {p:"src/**/*.ts"} ->lst
^ea ->f ^if {f.match:"*.test.ts"} {^ct} ^el {!rd @fs {p:f}}
```

### After (v0.2 — direct negation)

```
?fnd @fs {p:"src/**/*.ts" p:!~"*.test.ts" p:!~"*node_modules*"} ->lst
^ea ->f {!rd @fs {p:f}}
```

### Combined include/exclude pattern

```
?fnd @fs {p:"src/**" ext:[ts,tsx] ext:!~[test.ts,spec.ts] rgx:"export" rgx:!~"deprecated"} ->lst lmt:50
```

Reads: find in `src/**`, extensions `ts`/`tsx`, excluding `test.ts`/`spec.ts`, containing `export`, not containing `deprecated`, limit 50.

### Control flow negation

```
-- skip empty files
^ea {src:files} ->f <<
  ?rd @fs {p:f} ->content
  ^if neg {content.sz:0} <<
    !prs @mem {src:content}
  >>
>>
```

### L3 integration

In L3, negation is expressed with `!` prefix on a field:

```
Q.fs.fnd.src/**/*.ts.!*.test.ts.rec
```

### Grammar additions

```
param_value := '!~' value | value
modifier := ... | 'neg'
condition := 'neg'? '{' expr '}'
l3_neg_field := '!' raw_field | '!' backtick_field
```

### Vocabulary table update

Add to **Modifiers**:

| Token | Meaning |
|-------|---------|
| `neg` | negate / exclude |

## Summary of Changes

| Issue | Token(s) Added | Affects |
|-------|---------------|---------|
| Block scoping | `<<` `>>` | L2 control flow |
| Period escaping | `` ` `` (backtick) | L3 fields |
| Negation | `!~` (param prefix), `neg` (modifier) | L2 params, L2 control flow, L3 fields |

All three are additive. No existing v0.1 syntax is redefined or removed.
